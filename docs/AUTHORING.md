# Authoring guide — prebuilt assets

This guide covers how to author and ship the four categories of bundled asset that pathwise serves at startup: component libraries, facility-template libraries, value chains, and example workbooks. Read this if you are adding a new sector, updating a cost dataset, or building a new worked example.

**Prerequisite**: read [ARCHITECTURE.md](ARCHITECTURE.md) (storage section) before this document. The storage table there explains which format lives where and why.

---

## 1. Overview

Pathwise ships four categories of bundled asset, each in a distinct directory:

| Asset type | Source format | Directory | Loaded by |
|---|---|---|---|
| Component library seeds | JSON (`.json`) | `src/pathwise/assets/component_libraries/` | `GET /api/component-libraries` on first run |
| Facility-template libraries | JSON (`.json`) | `src/pathwise/assets/library/` | `GET /api/library/{lib_id}` |
| Value-chain specs + stage workbooks | JSON (`.json`) | `src/pathwise/assets/value_chains/` | `GET /api/value-chains` |
| Example workbooks | SQLite (`.sqlite`) | `src/pathwise/assets/examples/` | `POST /api/session/{sid}/example/{id}` |

**JSON = immutable bundled source, authored by hand.**
**SQLite = mutable working/persistence layer.** Example workbooks are SQLite so any SQLite viewer can inspect them; they are generated by Python scripts and committed. Component library seeds are JSON; they are automatically converted to SQLite in `data/component_libraries/` on the first API call and never modified in place.

Every directory that acts as an index has an `index.json` file listing the available entries. Adding a new asset without updating `index.json` means the UI will not find it.

See [API.md](API.md) for the full API reference.

## 2. The workbook model

A workbook is a Python `dict[str, list[dict[str, Any]]]` — a mapping from sheet name to a list of row dicts. On disk (SQLite) each sheet is a table; in memory it is just a list of rows.

`data/sheets.py` is the **single source of truth for sheet names**. Import the string constants from there rather than repeating raw strings. `data/schema.py` documents every column (label, type, description) and drives the frontend grid.

The five sheets required for a minimal runnable model are:

```
periods, commodities, technologies, processes, demand
```

### Core sheet reference

| Sheet | Purpose |
|---|---|
| `periods` | Modelled years (`year`, `duration_years`). The horizon is the union of all years in this sheet. |
| `commodities` | Stream definitions (`commodity_id`, `kind`, `unit`, optional `price`/`sale_price`). `kind` must be one of `energy`, `material`, `indirect`, `product`, `byproduct`. |
| `commodity_properties` | Free-form physical stream properties — `commodity_id`, `property`, `value` (e.g. `temperature_C`, `voltage_kV`, `pressure_bar`, `lhv_MJ_per_kg`). Carried as metadata (shown in the UI, preserved on round-trip); the optimiser does not constrain on them yet. |
| `impacts` | Environmental impact definitions (`impact_id`, `unit`). |
| `technologies` | Technology recipes (`technology_id`, `lifespan`, `capex`, `opex`, optional `introduction_year`/`phase_out_year`). |
| `io` | Unified input/output/impact table (`technology_id`, `target`, `role`, `coefficient`). `role` is `input`, `output`, or `impact`. |
| `processes` | Facility instances (`process_id`, `company`, `baseline_technology`, `capacity`). |
| `transitions` | Technology switch options (`from_technology`, `to_technology`, `action`, `capex_per_capacity`). `action` is `replace` or `renew`. |
| `markets` | Commodity supply or ETS markets (`market_id`, `target`, `price`, `sell_price`, etc.). |
| `storage` | Inter-year commodity stores (`storage_id`, `commodity_id`, etc.). |
| `demand` | Product demand requirements (`company`, `commodity_id`, `year`, `amount`). |
| `impact_caps` | Per-year environmental caps (`company`, `impact_id`, `year`, `limit`, `soft`, `penalty`). |
| `measures` | Retrofit measure definitions (`measure_id`, `type`, `target`, `facility` or `technology`, `lifetime`). |
| `measure_blocks` | Piecewise cost-curve steps per measure (`measure_id`, `block`, `reduction`, `capex`, `opex`). |
| `maccs` | MACC bundle membership (`macc`, `measure_id`). |
| `macc_links` | MACC deployment targets (`macc`, `facility`\|`technology`\|`commodity`\|`storage`). |
| `nodes` | Recursive hierarchy nodes (`node_id`, `parent_id`, `kind`, `level`, `label`). |
| `machines` | Leaf-machine details (`machine_id`, `baseline_technology`, `capacity`). |
| `connections` | Commodity flows between sibling nodes (`from_node`, `to_node`, `commodity_id`, `lag_years`). |
| `edges` | Commodity flows between flat processes (`from_process`, `to_process`, `commodity_id`). |

## 3. Temporal (year-varying) values

**Every numeric model parameter can be given as a scalar OR a per-year trajectory** — prices, costs (capex/opex/renewal/transition/storage), I/O coefficients (intensity, yield, emission factor), blend/slate share bounds, must-run floor, outage rate, edge capacity, market volume caps, storage efficiencies, demand, caps, budgets, and measure reduction. Supply the scalar on the entity's own row for a constant; add a row to the matching temporal sheet below to make it vary by year. The only intentionally scalar (structural) values are calendar-year markers (`introduction_year`, `phase_out_year`, `introduced_year`, `decommission_year`, `available_from`/`available_to`), asset lifetimes (`lifespan`, `lifetime`), the period's own `duration_years`, and a store's one-time `max_capacity` / `initial_level`.

Two conventions exist side by side. Choose one per attribute; the PyPSA-wide format wins when both are present for the same entity.

### 3a. Wide temporal sheets (`<entity>_t__<attr>`)

Sheet name convention: `<entity_base>_t__<attribute>` (double underscore). Each sheet has a `year` column plus one column per entity id. A blank cell means "use the scalar default for that year."

| Sheet | Applies to | Attribute |
|---|---|---|
| `commodities_t__price` | `commodity_id` | External purchase price |
| `commodities_t__sale_price` | `commodity_id` | Sale price |
| `commodities_t__max_purchase` | `commodity_id` | External purchase cap |
| `impacts_t__price` | `impact_id` | Carbon/ETS price |
| `technologies_t__capex` | `technology_id` | Replacement CAPEX |
| `technologies_t__opex` | `technology_id` | Fixed O&M |
| `technologies_t__renewal` | `technology_id` | Renewal cost |
| `technologies_t__min_capacity_factor` | `technology_id` | Must-run floor |
| `processes_t__capacity` | `process_id` | Capacity (nameplate throughput) |
| `processes_t__fixed_opex` | `process_id` | Fixed O&M override |
| `processes_t__failure_rate` | `process_id` | Forced-outage fraction |
| `markets_t__price` | `market_id` | Market buy price |
| `markets_t__sell_price` | `market_id` | Market sell price |
| `markets_t__allocation` | `market_id` | ETS free allocation |
| `markets_t__max_buy` | `market_id` | Buy volume cap |
| `markets_t__max_sell` | `market_id` | Sell volume cap |
| `storage_t__capex_per_capacity` | `storage_id` | Store build cost (year-0 value used) |
| `storage_t__fixed_opex_per_capacity` | `storage_id` | Store fixed O&M |
| `storage_t__charge_efficiency` | `storage_id` | Charge efficiency |
| `storage_t__discharge_efficiency` | `storage_id` | Discharge efficiency |
| `storage_t__standing_loss` | `storage_id` | Standing loss |
| `investment_budget_t__limit` | `budget_id` | Investment budget |
| `min_production_t__amount` | `min_id` | Minimum production floor |
| `demand_t__amount` | `demand_id` | Demand target |
| `impact_caps_t__limit` | `cap_id` | Impact cap |

Example — an electricity price that rises 2025→2030:

```json
{ "commodities_t__price": [
    { "year": 2025, "electricity": 80.0 },
    { "year": 2030, "electricity": 95.0 }
]}
```

### 3b. Long-format temporal sheets

Used for multi-keyed values where a wide matrix would be sparse or ambiguous.

| Sheet | Key columns | Value column(s) | Purpose |
|---|---|---|---|
| `commodity_prices` | `commodity_id`, `year` | `price`, `sale_price` | Per-commodity price trajectory |
| `technologies_prices` | `technology_id`, `year` | `capex`, `opex` | Per-technology cost trajectory (emitted by the component library) |
| `io_t` | `technology_id`, `target`, `role`, `year` | `coefficient`, `share_min`, `share_max` | Year-varying I/O intensity / yield / emission factor, and blend/slate share bounds |
| `transitions_t` | `from_technology`, `to_technology`, `year` | `capex_per_capacity` | Year-varying transition CAPEX |
| `edges_t` | `from_process`, `to_process`, `commodity_id`, `year` | `max_flow` | Year-varying edge capacity |
| `commodity_impacts_t` | `commodity_id`, `impact_id`, `year` | `factor` | Year-varying upstream carbon intensity |
| `measure_blocks_t` | `measure_id`, `block`, `year` | `capex`, `opex`, `reduction` | Year-varying block costs (absolute, scaled to instance capacity) and abatement |
| `impact_prices` | `impact_id`, `year` | `price` | Carbon price trajectory (long-format alternative to `impacts_t__price`) |
| `market_prices` | `market_id`, `year` | `price`, `sell_price`, `allocation` | Market price trajectory (long-format alternative to `markets_t__*`) |

### Interpolation rule

Sparse year points are **linearly interpolated** onto the full model horizon. Before the earliest known year the value is flat-held at the first point; after the latest known year it is flat-held at the last point. A trajectory overrides the scalar for every modelled year — you do not need to supply a value for every year in `periods`.

Example — a capex that falls from 1 000 in 2025 to 600 in 2035 (interpolated at 2030 = 800):

```json
{ "technologies_prices": [
    { "technology_id": "Electrolyser", "year": 2025, "capex": 1000.0 },
    { "technology_id": "Electrolyser", "year": 2035, "capex":  600.0 }
]}
```

The assembler in `data/assemble.py` calls `data/trajectory.py:interpolate()` for every trajectory before handing values to the optimiser.

## 4. Authoring a component library

A **component library** (`ComponentLibrary` in `data/components.py`) is the editable palette of reusable building blocks: commodities (streams), technologies (recipes), measures, and MACC bundles. The Value-Chain builder draws from it; placing a technology stamps it as a machine node.

### 4a. Field reference

**`ComponentLibrary`**

| Field | Type | Required | Notes |
|---|---|---|---|
| `label` | `str` | no | Human-readable name shown in the UI. |
| `commodities` | `list[CommodityTemplate]` | no | Streams the technologies use. |
| `technologies` | `list[TechnologyTemplate]` | no | Technology recipes. |
| `measures` | `list[MeasureTemplate]` | no | Individual retrofit measures. |
| `maccs` | `list[MaccGroup]` | no | Named bundles of measures. |
| `machines` | `list[MachineComponent]` | no | Legacy — not authored by the builder; round-trips older files. |
| `groups` | `list[GroupComponent]` | no | Legacy — not authored by the builder; round-trips older files. |
| `notes_by_sector` | `dict[str, str]` | no | Freetext notes keyed by sector name; optimiser ignores them. |

**`CommodityTemplate`**

| Field | Type | Notes |
|---|---|---|
| `commodity_id` | `str` | Unique id. |
| `kind` | `str` | One of `energy`, `material`, `indirect`, `product`, `byproduct`. |
| `unit` | `str` | Physical unit string (e.g. `"GJ"`, `"t"`, `"MWh"`). |
| `price` | `float \| None` | Scalar external purchase price. |
| `sale_price` | `float \| None` | Scalar sale price. |
| `price_by_year` | `dict[int, float]` | Year→price trajectory; overrides `price` when non-empty. |
| `sale_price_by_year` | `dict[int, float]` | Year→sale_price trajectory. |
| `sector` | `str \| None` | Owning sector label for grouping in the Component builder; optimiser ignores it. |
| `notes` | `str` | Freetext; optimiser ignores it. |

**`TechnologyTemplate`**

| Field | Type | Notes |
|---|---|---|
| `technology_id` | `str` | Unique id. |
| `lifespan` | `int` | Asset life in years (default 20). |
| `capex` | `float` | Replacement CAPEX per unit capacity (default 0). |
| `opex` | `float` | Fixed O&M per unit throughput (default 0). |
| `capex_by_year` | `dict[int, float]` | Trajectory overriding `capex`. |
| `opex_by_year` | `dict[int, float]` | Trajectory overriding `opex`. |
| `introduction_year` | `int \| None` | First year it may be adopted. |
| `phase_out_year` | `int \| None` | Last year it may operate. |
| `io` | `list[IoRow]` | Inputs, outputs, and impacts (min 1 row). |
| `maccs` | `list[str]` | IDs of `MaccGroup` entries that apply to this technology. |
| `notes` | `str` | Freetext. |

**`IoRow`** (one row of the `io` sheet)

| Field | Type | Notes |
|---|---|---|
| `target` | `str` | Commodity id (input/output) or impact id (impact). |
| `role` | `str` | `"input"`, `"output"`, or `"impact"`. |
| `coefficient` | `float` | Per-unit-throughput amount (intensity or yield or factor). |
| `is_product` | `bool` | True if this output meets demand (default false). |
| `group` | `str \| None` | Blend group name — inputs with the same group compete in a share-bounded mix. |
| `share_min` | `float \| None` | Minimum share within the blend group (0–1). |
| `share_max` | `float \| None` | Maximum share within the blend group (0–1). |

**`MeasureTemplate`**

| Field | Type | Notes |
|---|---|---|
| `measure_id` | `str` | Unique id. |
| `label` | `str` | Display name. |
| `type` | `str` | One of `energy_efficiency`, `emission_reduction`, `environmental`. |
| `target` | `str` | Commodity id (`energy_efficiency`) or impact id (other types). |
| `lifetime` | `int` | Measure asset life in years (default 15). |
| `blocks` | `list[MeasureBlockTemplate]` | Piecewise cost-curve steps (min 1). |
| `notes` | `str` | Freetext. |

**`MeasureBlockTemplate`**

| Field | Type | Notes |
|---|---|---|
| `reduction` | `float` | Fraction of the target reduced at full adoption (0 < x ≤ 1). |
| `capex_per_capacity` | `float` | CAPEX per unit of facility capacity. Scaled to instance capacity on placement. |
| `opex_per_capacity` | `float` | Fixed O&M per unit capacity per year. |
| `capex_per_capacity_by_year` | `dict[int, float]` | Trajectory overriding `capex_per_capacity`. |
| `opex_per_capacity_by_year` | `dict[int, float]` | Trajectory overriding `opex_per_capacity`. |

**`MaccGroup`**

| Field | Type | Notes |
|---|---|---|
| `macc_id` | `str` | Unique id. |
| `label` | `str` | Display name. |
| `measures` | `list[str]` | Measure ids belonging to this bundle. |
| `notes` | `str` | Freetext. |

### 4b. Minimal JSON example

```json
{
  "label": "Example library",
  "commodities": [
    { "commodity_id": "natural_gas", "kind": "energy", "unit": "GJ", "price": 8.0 },
    { "commodity_id": "heat",        "kind": "product",  "unit": "GJ" }
  ],
  "technologies": [
    {
      "technology_id": "Boiler",
      "lifespan": 20,
      "capex": 120.0,
      "opex": 5.0,
      "io": [
        { "target": "natural_gas", "role": "input",  "coefficient": 1.05 },
        { "target": "heat",        "role": "output", "coefficient": 1.0, "is_product": true },
        { "target": "CO2",         "role": "impact", "coefficient": 0.056 }
      ],
      "maccs": ["boiler_eff"]
    }
  ],
  "measures": [
    {
      "measure_id": "insulation",
      "label": "Pipe insulation",
      "type": "energy_efficiency",
      "target": "natural_gas",
      "lifetime": 15,
      "blocks": [
        { "reduction": 0.05, "capex_per_capacity": 10.0, "opex_per_capacity": 0.2 }
      ]
    }
  ],
  "maccs": [
    { "macc_id": "boiler_eff", "label": "Boiler efficiency", "measures": ["insulation"] }
  ]
}
```

### 4c. Runtime loading

Component library seeds (JSON files in `src/pathwise/assets/component_libraries/`) are automatically converted to SQLite on the first call to `GET /api/component-libraries` and written to `data/component_libraries/<stem>.sqlite`. That directory is writable; the seed directory is read-only.

Config variable: `PATHWISE_COMPONENT_SEEDS_DIR` (default: the bundled `assets/component_libraries/`).

No `index.json` is needed for component libraries — the router enumerates all `.sqlite` files in the writable directory.

### 4d. How technologies, commodities, measures, and MACCs relate

1. A `MeasureTemplate` targets a single commodity or impact and has one or more cost-curve blocks.
2. A `MaccGroup` collects measure ids into a named bundle.
3. A `TechnologyTemplate` lists the MACC bundle ids that apply to it (`maccs` field).
4. When the Value-Chain builder places a technology as a machine node, every measure of every linked MACC is stamped onto that machine, with block costs scaled to the instance capacity (`capex = capex_per_capacity × capacity`).

## 5. Authoring a facility-template library

A **facility-template library** (`Library` in `data/library.py`) is a static, read-only catalogue of prebuilt facility archetypes and process chains. Users insert them wholesale into a model from the Library panel, rather than building cross-referenced tables by hand.

### 5a. Field reference

**`Library`**

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | `str` | yes | Matches the file stem and the `index.json` entry id. |
| `label` | `str` | yes | Display name. |
| `commodities` | `list[CommodityTemplate]` | no | Streams the facilities use. |
| `facilities` | `list[FacilityTemplate]` | yes (min 1) | The facility archetypes. |
| `chains` | `list[ChainTemplate]` | no | Multi-stage process chains. |

**`FacilityTemplate`**

| Field | Type | Required | Notes |
|---|---|---|---|
| `facility_id` | `str` | yes | Unique id within the library. |
| `label` | `str` | yes | Display name. |
| `description` | `str` | no | Short description. |
| `technology` | `TechnologyTemplate` | yes | The baseline technology (recipe + costs). |
| `alternatives` | `list[Alternative]` | no | Lower-carbon options the facility may transition to. |
| `measures` | `list[MeasureTemplate]` | no | Retrofit measures stamped on every instance. |
| `default_capacity` | `float` | no | Capacity used when instantiating (must be > 0; default 1000). |
| `source` | `SourceRef` | **yes** | Citation — mandatory; an uncited entry is a validation error. |

**`Alternative`**

| Field | Type | Notes |
|---|---|---|
| `technology` | `TechnologyTemplate` | The alternative technology (its own recipe + costs). |
| `transition_capex_per_capacity` | `float` | CAPEX to switch into this alternative per unit capacity. |

**`SourceRef`** — mandatory on every `FacilityTemplate` and `ChainTemplate`

| Field | Type | Notes |
|---|---|---|
| `name` | `str` | Source title (min 3 chars). |
| `url` | `str` | Must start with `http://` or `https://`. A validation error is raised if absent or non-http. |
| `year` | `int` | Publication/data year (1900–2100). |
| `region` | `str` | Geographic basis (e.g. `"global"`, `"KR"`, `"EU"`; default `"global"`). |
| `basis` | `str` | What the numbers represent (default `"indicative"`). |
| `notes` | `str \| None` | Optional caveats. |

**`ChainTemplate`**

| Field | Type | Required | Notes |
|---|---|---|---|
| `chain_id` | `str` | yes | Unique id within the library. |
| `label` | `str` | yes | Display name. |
| `description` | `str` | no | |
| `stages` | `list[ChainStage]` | yes (min 1) | Ordered stages. |
| `demand_hint` | `DemandHint \| None` | no | Suggested demand when the chain is instantiated. |
| `source` | `SourceRef` | **yes** | Mandatory citation. |

**`ChainStage`**

| Field | Type | Notes |
|---|---|---|
| `facility` | `str` | A `facility_id` in this library. |
| `feeds` | `list[str]` | Upstream `facility_id` values whose outputs this stage consumes. The assembler derives `edges` from the shared commodity. |

```json
      "facility_id": "boiler",

        "capex": 120.0,
        "opex": 5.0,
        "io": [
          { "target": "natural_gas", "role": "input",  "coefficient": 1.05 },
          { "target": "heat",        "role": "output", "coefficient": 1.0, "is_product": true }
        ]
      },
      "alternatives": [],
      "default_capacity": 200.0,
      "source": {
        "name": "IEA Energy Technology Perspectives 2023",
        "url": "https://www.iea.org/reports/energy-technology-perspectives-2023",
        "year": 2023,
        "region": "global",
        "basis": "indicative"
      }
    }
  ]
}
```

### 5c. index.json

`src/pathwise/assets/library/index.json` is a JSON array. Each entry must have at minimum `id`, `label`, and `file`:

```json
[
  {
    "id": "heat",
    "label": "District heat",
    "file": "heat.json",
    "description": "1 facility template(s)"
  }
]
```

Config variable: `PATHWISE_LIBRARY_DIR` (default: `src/pathwise/assets/library/`).

API route: `GET /api/library` lists entries; `POST /api/session/{sid}/library` inserts a facility or chain into the session workbook.

## 6. Authoring a value chain

A **value chain** (`ValueChainSpec` in `data/valuechain.py`) links several pathwise models (stages) into a directed acyclic graph. An upstream stage's solved price, marginal price, carbon intensity, or produced volume is injected into the downstream stage's inputs before it solves.

### 6a. Field reference

**`ValueChainSpec`**

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | `str` | yes | Matches the file stem and `index.json` entry. |
| `label` | `str` | no | Display name. |
| `stages` | `list[Stage]` | yes (min 1) | The stage-models to solve. |
| `links` | `list[CouplingLink]` | no | Directed couplings between stages. Active links must form a DAG. |

**`Stage`**

| Field | Type | Notes |
|---|---|---|
| `id` | `str` | Unique id within the chain; must not repeat. |
| `label` | `str` | Display name. |
| `model` | `str` | Path to the stage workbook, relative to the value-chain spec file (e.g. `"models/electricity.json"`). |
| `region` | `str` | Optional region label (e.g. `"KR"`). |
| `sector` | `str` | Optional sector label (e.g. `"electricity"`). |
| `scenario` | `dict` | Optional per-stage scenario overrides deep-merged onto the run scenario. |

**`CouplingLink`**

| Field | Type | Notes |
|---|---|---|
| `from_stage` | `str` | Upstream stage id. |
| `to_stage` | `str` | Downstream stage id. |
| `commodity` | `str` | The shared commodity the signal is about (e.g. `"electricity"`). |
| `signals` | `list[str]` | Subset of `["price", "marginal_price", "carbon_intensity", "volume"]`. |
| `impact` | `str` | Which impact `carbon_intensity` refers to (default `"CO2"`). |
| `lag_years` | `int` | Years to shift the signal forward (default 0, must be ≥ 0). |
| `feedback` | `bool` | If true, the downstream stage's consumption feeds back as upstream demand (resolved by fixed-point iteration when `iterations > 1`; default false). |
| `alternative_of` | `str \| None` | If set, this link is an alternative supply choice for the same downstream commodity as the named link. Alternative links do not participate in the forward cascade automatically. |

### 6b. Signals

| Signal | What transfers |
|---|---|
| `price` | Average-cost proxy from the upstream stage for the shared commodity. |
| `marginal_price` | True marginal cost (finite-difference on demand). Takes precedence over `price` when both are present. |
| `carbon_intensity` | Upstream emissions per unit of the commodity (for the named `impact`). Written to `commodity_impacts_t` in the downstream workbook. |
| `volume` | Upstream produced quantity written to `commodities_t__max_purchase` in the downstream workbook — caps how much the downstream stage can buy externally. |

### 6c. Value-chain JSON example

```json
{
  "id": "elec_steel",
  "label": "Electricity → Steel (KR)",
  "stages": [
    {
      "id": "electricity",
      "label": "Electricity (KR)",
      "model": "models/electricity.json",
      "region": "KR",
      "sector": "electricity"
    },
    {
      "id": "steel",
      "label": "Steel (KR)",
      "model": "models/steel.json",
      "region": "KR",
      "sector": "steel"
    }
  ],
  "links": [
    {
      "from_stage": "electricity",
      "to_stage": "steel",
      "commodity": "electricity",
      "signals": ["price"],
      "lag_years": 0
    }
  ]
}
```

### 6d. Stage workbooks

Each stage's `model` path is resolved relative to the value-chain JSON file. The convention is to place them in a `models/` subdirectory. Stage workbooks are plain pathwise workbooks (JSON) containing at minimum `periods`, `commodities`, `technologies`, `processes`, and `demand`.

### 6e. index.json

`src/pathwise/assets/value_chains/index.json` is a JSON array. Each entry must have at minimum `id`, `label`, `file`, and `description`:

```json
[
  {
    "id": "elec_steel",
    "label": "Electricity → Steel (KR)",
    "file": "elec_steel.json",
    "description": "..."
  }

```

Config variable: `PATHWISE_VALUE_CHAINS_DIR` (default: `src/pathwise/assets/value_chains/`).

### 6f. DAG constraint

The active links (those with `alternative_of == null`) must form a directed acyclic graph. `ValueChainSpec.order()` runs Kahn's algorithm on construction and raises `ValueError` if a cycle is found. Test this before committing by importing and constructing the spec:

```python
from pathwise.data.valuechain import load_value_chain
load_value_chain("src/pathwise/assets/value_chains/my_chain.json")  # raises on cycle
```

## 7. Building an example workbook (.sqlite)

Example workbooks are SQLite files committed to `src/pathwise/assets/examples/`. They are generated by Python scripts (see `scripts/build_green_steel_example.py`) rather than authored by hand because they embed solved-baseline values (e.g. sizing a CO₂ cap from unconstrained throughput).

### 7a. Pattern

1. **Build the workbook dict in Python** — construct the `{sheet: [row, …]}` dict, either by calling `instantiate()` on a `ComponentLibrary` or by directly populating the required sheets.
2. **Add scenario sheets** — `periods`, `impact_prices`, `markets`, `impact_caps`, `demand`, `transitions`, and any temporal trajectories.
3. **Optionally solve to calibrate** — the build script runs `run_model()` on an uncapped baseline to derive realistic cap values, then asserts the constrained solve is optimal.
4. **Write to SQLite** using `pathwise.api.workbook_io.write_sqlite(workbook)`.
5. **Commit both** the script and the resulting `.sqlite` file.

### 7b. Minimal build script skeleton

```python
from pathwise.api.workbook_io import write_sqlite
from pathwise.data.components import ComponentLibrary, instantiate

YEARS = [2025, 2030, 2035, 2040, 2045, 2050]

lib = ComponentLibrary.model_validate(MY_LIBRARY_DICT)
wb = instantiate(lib, "my_top_group", instance_id="root")

wb["periods"] = [{"year": y, "duration_years": 5} for y in YEARS]
wb["demand"] = [
    {"company": "root/my_company", "commodity_id": "steel", "year": y, "amount": 1_000_000}
    for y in YEARS
]

output_path = "src/pathwise/assets/examples/my_example.sqlite"
open(output_path, "wb").write(write_sqlite(wb))
```

Run with `uv run python scripts/my_build_script.py`.

### 7c. index.json

`src/pathwise/assets/examples/index.json` is a JSON array. Each entry must have `id`, `label`, `file`, and `description`. The optional `library` field names a component library seed (by id) that a "Load with library" action will pre-load alongside the example:

```json
[
  {
    "id": "my_example",
    "label": "My sector — region",
    "file": "my_example.sqlite",
    "library": "my_lib",
    "description": "One sentence."
  }
]
```

Config variable: `PATHWISE_EXAMPLES_DIR` (default: `src/pathwise/assets/examples/`).

API route: `GET /api/examples` returns the index; `POST /api/session/{sid}/example/{id}` loads the workbook into the session.

```bash
uv run pytest tests/data/test_library.py tests/data/test_components.py tests/api/test_component_libraries.py -v
```

Or run the full suite:

```bash
uv run pytest
```

If you add a new library file, add it to `index.json` and ensure the test parametrisation (`LIBRARY_FILES = sorted(p for p in LIB_DIR.glob("*.json") ...`) picks it up automatically — no test edit is needed.

If you add a new example workbook, add an entry to `examples/index.json`. The example workbooks are not auto-validated by CI (they are SQLite), but the build script that generates them should assert both a system-scope and a value-chain-scope solve succeed before writing the file.

## 9. Optimisation methods (backends)

The optimisation *method* is a user choice in **Settings → Optimisation method**, populated from `/api/config`'s `backends` list. Each backend implements the same `run(model, scenario, options)` contract but solves a different problem:

| Backend (`name`) | Label | What it does |
|---|---|---|
| `linopy` | linopy + HiGHS | The default MILP: co-optimises the whole fleet/network over the horizon. |
| `portfolio` | Portfolio (risk vs reward) | Frames candidate transitions as portfolio assets and allocates by risk/reward. |
| `macc` | MACC (greedy abatement) | Greedy marginal-cost deployment against an emission target — **not** an LP. |

The run path passes the selected method through verbatim (`options.backend`), so a workbook authored for one method is run by whichever method is chosen — the backend reads only the sheets it needs.

### 9a. The MACC (greedy abatement) mode

This mode reproduces the marginal-abatement-cost runners common in sector decarbonisation models (e.g. the Korean petrochemical MACC): rank abatement levers by `$/tCO2`, then each year deploy the cheapest first until the year's required abatement (`BAU − target`) is met or the curve is exhausted. Deployment is **irreversible** (carried forward), so once option potentials saturate, residual emissions can drift above an ever-tightening target — the model does not force the target, it chases it.

It is **sector-agnostic**: all the cost/potential maths live in your build script, which emits three sheets the backend consumes verbatim.

| Sheet | Columns | Meaning |
|---|---|---|
| `macc_target` | `year`, `bau`, `target` | The emission line to chase. `required = max(0, bau − target)`. Pre-interpolate to every modelled year. |
| `macc_curve` | `option_id`, `year`, `potential`, `cost`, `capex` | One row per (option, year). `potential` = max abatement available that year; `cost` = the `$/abated` **ranking key** (cheapest first); `capex` = cost **booked per unit abated** (drives cumulative CAPEX). |
| `macc_options` | `option_id`, `label`, `available_from` | Optional metadata. `available_from` gates an option to a first year. |

Algorithm, per year (ascending):

```
required  = max(0, bau − target)
rows      = curve rows for year with potential>0 and year≥available_from, sorted by cost asc
remaining = max(0, required − Σ deployed)          # carry-forward (irreversible)
for option in rows:                                 # cheapest first
    add = min(remaining, potential − deployed[option])   # ≥ 0, capped at potential
    deployed[option] += add; remaining −= add
    cumulative_capex += add × capex
actual_emissions = bau − Σ deployed
```

The result carries an `outputs.macc` block (`by_year`, `options`, `cumulative_capex`); `objective` is the final-year cumulative CAPEX; the residual emission path is surfaced under `summary.impacts`. Two unit-conversion subtleties belong in the build script, **not** the backend: bake any annualisation/lifetime factor into the `capex` column (so `Σ add × capex` lands in the units you report), and resolve any mutually-exclusive option choice (e.g. pick one of two carriers) by emitting curve rows only for the chosen option.

See `src/pathwise/backends/macc_backend.py` and `tests/backends/test_macc_backend.py` for the reference implementation and a hand-computed regression test.
